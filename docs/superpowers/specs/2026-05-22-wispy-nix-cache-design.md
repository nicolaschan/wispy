# wispy — design

A drop-in GitHub Action that uses a Cloudflare R2 bucket as a Nix binary
cache, so a CI job's nix builds can be substituted from the cache on hit
and pushed back to the cache on miss. Implemented as a TypeScript GitHub
Action.

This document is the design contract. It locks the architecture, the
public interface, the failure semantics, and the boundaries of v1. The
implementation plan will be derived from it.

## Goals

- A workflow author adds one `uses:` line and six inputs and gets
  substantial speedups on subsequent runs.
- All locally-built store paths are pushed to R2 transparently in the
  background, with negligible wall-clock impact on the build itself.
- A failure in any part of the cache pipeline never fails the user's
  build.
- The trust model (signing keys, substituter URLs) is portable to a
  future Cloudflare Worker that fronts R2 — the worker is a one-line
  config change for users, not a re-architecture.

## Non-goals (v1)

- Garbage collection, retention, or TTL of cache entries. Users manage
  this with R2 lifecycle rules.
- Per-branch or per-PR cache namespacing.
- Mirroring `cache.nixos.org` into R2.
- Public/unauthenticated read access. The bucket stays private; reads
  use S3 credentials via Nix's native `s3://` substituter.
- Cross-run retry of failed uploads.
- macOS runner support. Linux x86_64 and aarch64 only. macOS fails fast
  with a clear "not supported in v1" message.
- A daemon-mediated pipeline like `magic-nix-cache`. The future
  Cloudflare Worker takes that role; building it now duplicates effort.

## Architecture overview

The action runs in three phases inside a single workflow job. It assumes
Nix is already installed by a separate prior step (e.g.
`DeterminateSystems/nix-installer-action`).

```
┌────────────────────────────────────────────────────────────────────┐
│ 1. nix-installer-action                  (user's step, not ours)   │
│ 2. wispy/setup-cache@v1  main:                                     │
│        - validate inputs                                           │
│        - write signing private key to a 0600 file                  │
│        - append wispy config block to /etc/nix/nix.conf            │
│            * extra-substituters    (R2 s3:// URL)                  │
│            * extra-trusted-public-keys                             │
│            * secret-key-files      (so daemon signs at build time) │
│            * post-build-hook       (path to hook.sh)               │
│        - reload nix-daemon                                         │
│        - smoke-test R2 reachability                                │
│        - spawn detached `uploader` node process                    │
│        - record its PID                                            │
│ 3. nix build / flake check / …           (user's steps)            │
│ 4. wispy/setup-cache@v1  post:                                     │
│        - append sentinel line to queue file                        │
│        - wait for uploader PID to exit                             │
│        - read uploader's status file, set outputs, print summary   │
│        - remove wispy block from nix.conf (idempotent)             │
│        - shred signing key file                                    │
└────────────────────────────────────────────────────────────────────┘
```

### Process model

Setup creates a single runtime directory `$RUNNER_TEMP/wispy/` (mode
0755, where `$RUNNER_TEMP` is the per-job temp dir GitHub Actions
provides). All runtime files live there:

| File             | Mode | Owner   | Written by | Read by                |
| ---------------- | ---- | ------- | ---------- | ---------------------- |
| `signing.key`    | 0600 | runner  | setup      | nix-daemon (as root)   |
| `hook.sh`        | 0755 | runner  | setup      | nix-daemon (as root)   |
| `queue`          | 0666 | runner  | hook       | uploader               |
| `uploader.pid`   | 0644 | runner  | setup      | post step              |
| `status.json`    | 0644 | runner  | uploader   | post step              |

The hook is shell, not node. nix-daemon execs the hook program directly
for every built derivation; paying node startup cost on each derivation
would add real latency. The hook is generated by setup from a template
so that the queue path is baked in (nix-daemon does not inherit
`$RUNNER_TEMP`):

```sh
#!/usr/bin/env bash
# $RUNNER_TEMP/wispy/hook.sh — generated by setup, invoked by nix-daemon
printf '%s\n' "$OUT_PATHS" >> '<absolute path to queue file>' 2>/dev/null || true
```

The trailing `|| true` and stderr redirect guarantee the hook never
fails the build, even if the queue file disappears.

The uploader is a long-running node process spawned during the main
step as a detached child (`child_process.spawn(..., { detached: true,
stdio: ['ignore', <log fd>, <log fd>] })`, followed by `unref()`).
Setup writes the child's PID to `uploader.pid`.

The uploader tails the queue file by tracking a byte offset:

1. `fs.stat` the queue file.
2. If `size > lastOffset`, open at `lastOffset`, read to `size`,
   advance `lastOffset`, parse complete lines (buffer any trailing
   partial line for the next tick).
3. Sleep 250 ms, repeat.

250 ms is sufficient because build steps take seconds at minimum. This
pattern avoids `fs.watch`'s cross-platform reliability problems and
adds no dependencies. New paths are submitted to an in-process work
pool with bounded concurrency (`upload-concurrency`, default 8). Each
work unit is one `nix copy --to <s3-url> <path>` invocation. On reading
the sentinel line `__WISPY_EOF__`, the uploader stops tailing, awaits
all in-flight uploads, writes `status.json`, and exits 0.

### Daemon reload

After editing `/etc/nix/nix.conf`, setup runs
`sudo systemctl restart nix-daemon` (GHA Linux runners have
passwordless sudo and the DeterminateSystems installer registers
nix-daemon as a systemd unit). A reload is required because
`secret-key-files` and `post-build-hook` are daemon-side options — the
client re-reads nix.conf per invocation, but the daemon does not. A
non-zero exit from the restart command fails setup loudly: without the
reload, the hook never fires and the user silently gets no caching.

### Component responsibilities

| Component         | Language   | Job                                                                                       |
| ----------------- | ---------- | ----------------------------------------------------------------------------------------- |
| `action.yml`      | YAML       | composite-action manifest: inputs, `runs.steps`, `runs.post-steps`                        |
| `src/main.ts`     | TypeScript | entry for main step; orchestrates setup, key writing, nix.conf edit, daemon reload, uploader spawn |
| `src/post.ts`     | TypeScript | entry for post step; sentinel + wait + summary + cleanup                                  |
| `src/nixconf.ts`  | TypeScript | pure helpers: `applyWispyBlock(conf, block)`, `removeWispyBlock(conf)`. Testable in isolation. |
| `src/uploader.ts` | TypeScript | entry for the detached uploader process; polls queue, runs `nix copy` batches             |
| `src/r2.ts`       | TypeScript | construct the `s3://` substituter URL; perform the SigV4 reachability check via AWS SDK   |
| `src/inputs.ts`   | TypeScript | typed input parsing and validation                                                         |
| `scripts/hook.sh` | bash       | the only shell script: nix-daemon's post-build-hook target                                |

Each TypeScript module has one purpose and no shared mutable state. The
main and post entries are thin orchestrators that compose the modules.

## Public interface

### `action.yml` inputs

| Input                       | Required | Default                       | Description |
| --------------------------- | -------- | ----------------------------- | ----------- |
| `r2-bucket`                 | yes      | —                             | R2 bucket name |
| `r2-account-id`             | yes      | —                             | Cloudflare account ID; used to build the endpoint URL `https://<id>.r2.cloudflarestorage.com` |
| `r2-access-key-id`          | yes      | —                             | R2 token's access key ID |
| `r2-secret-access-key`      | yes      | —                             | R2 token's secret access key |
| `signing-private-key`       | yes      | —                             | Full contents of the ed25519 private key file |
| `signing-public-key`        | yes      | —                             | Public key string (e.g. `wispy-1:wxyz...`) |
| `upload-concurrency`        | no       | `8`                           | Maximum parallel `nix copy` invocations in the uploader |
| `extra-substituters`        | no       | `https://cache.nixos.org/`    | Space-separated list; merged into `extra-substituters` in nix.conf |
| `extra-trusted-public-keys` | no       | `cache.nixos.org-1:6Nm…IXLPMzw=` | Space-separated; merged into `extra-trusted-public-keys` |
| `skip-push`                 | no       | `false`                       | If `true`, pull is configured but push is not. Canonical usage: `${{ github.event.pull_request && github.event.pull_request.head.repo.fork }}` |

### Outputs

| Output         | When      | What |
| -------------- | --------- | ---- |
| `paths-pushed` | post step | count of unique store paths uploaded to R2 |
| `bytes-pushed` | post step | sum of compressed NAR bytes uploaded |
| `paths-failed` | post step | count of paths that failed to upload (build is not failed; this surfaces as a warning) |

### Example workflow

```yaml
name: build
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: DeterminateSystems/nix-installer-action@main
      - uses: nicolaschan/wispy@v1
        with:
          r2-bucket:            ${{ vars.WISPY_R2_BUCKET }}
          r2-account-id:        ${{ vars.WISPY_R2_ACCOUNT_ID }}
          r2-access-key-id:     ${{ secrets.WISPY_R2_ACCESS_KEY_ID }}
          r2-secret-access-key: ${{ secrets.WISPY_R2_SECRET_ACCESS_KEY }}
          signing-private-key:  ${{ secrets.WISPY_SIGNING_PRIVATE_KEY }}
          signing-public-key:   ${{ vars.WISPY_SIGNING_PUBLIC_KEY }}
          skip-push: ${{ github.event.pull_request && github.event.pull_request.head.repo.fork }}
      - run: nix build .#default
```

### One-time setup (README content)

```sh
# Generate signing keypair (do this once, locally)
nix-store --generate-binary-cache-key \
  "wispy-$(whoami)-1" /tmp/wispy.priv /tmp/wispy.pub

# Store in GitHub:
#   Settings → Secrets   → New: WISPY_SIGNING_PRIVATE_KEY  ← contents of /tmp/wispy.priv
#   Settings → Variables → New: WISPY_SIGNING_PUBLIC_KEY   ← contents of /tmp/wispy.pub
#     (Public key as Variable, not Secret, so it appears unredacted in logs.)

# Create R2 bucket and API token (Cloudflare dashboard or wrangler):
#   Bucket: wispy-cache
#   Token:  scoped Object Read & Write on that bucket only
#   Store creds in Secrets, account ID in a Variable
```

## Push pipeline

### Signing happens at build time, not upload time

`secret-key-files` is set in `nix.conf` so nix-daemon signs every locally
built store path as it's produced. The hook and uploader never touch
keys. The upload step (`nix copy`) only copies already-signed NAR and
narinfo. This means the secret key lives in exactly one place on disk
(the 0600 file referenced by nix.conf), with one consumer (nix-daemon).

### Queue protocol

The hook appends one line per built derivation. nix's `$OUT_PATHS` is a
space-separated list of paths produced by that derivation. The hook
writes the line verbatim:

```
/nix/store/abc...-foo /nix/store/def...-foo-doc
/nix/store/ghi...-bar
__WISPY_EOF__
```

The uploader splits each non-sentinel line on whitespace, dedups across
lines, and submits each unique path to its work pool. On reading the
sentinel, the uploader stops reading, waits for in-flight uploads to
finish, writes `status.json`, and exits.

The hook is best-effort. If the queue file is unwritable, the hook logs
to stderr and exits 0 — never failing the build. Setup creates the
queue file with `chmod 666` up front; failure to create it during setup
is a fatal setup error (see Failure modes below).

### Concurrency

The uploader runs up to `upload-concurrency` `nix copy` invocations in
parallel (default 8). `nix copy` itself is single-threaded per
invocation but checks for object existence before uploading, so
concurrent jobs from multiple workflow runs writing to the same bucket
are safe and idempotent.

### Why a long-running uploader instead of inline async in the hook

Backgrounding `nix copy` from the hook (`nix copy ... &`) detaches the
upload from the nix-daemon-spawned hook process. Those backgrounded
processes orphan to PID 1, with no way for the post step to know when
they finish. Either the post step exits while uploads are mid-flight
(losing them when the runner shuts down) or it has to scan `/proc` for
zombies, which is brittle. A single uploader process owned by the
action makes the drain trivial: write sentinel, wait on a known PID.

## Pull pipeline

The pull side is static configuration only. Setup appends a single
substituter URL and a single trusted public key to nix.conf using the
`extra-substituters` / `extra-trusted-public-keys` directives (these
merge with existing values rather than overwriting them).

The substituter URL takes the form:

```
s3://<bucket>?endpoint=https://<account-id>.r2.cloudflarestorage.com&region=auto
```

Credentials are passed to nix via `AWS_ACCESS_KEY_ID` and
`AWS_SECRET_ACCESS_KEY` environment variables set in the nix-daemon's
environment by the setup step (or, equivalently, in `~/.aws/credentials`
for the runner user; we'll use the env-var path because it's more
auditable in a CI context).

No nix daemon code path needs to know about wispy. `nix build` resolves
substituters as configured and pulls/verifies signed paths normally.

## Failure modes

The principle: cache problems must never fail the build. The build is
the product.

### Push-side runtime failures

| Failure                                       | What happens                            | Action's response                                                                                       |
| --------------------------------------------- | --------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| R2 returns 5xx mid-upload                     | `nix copy` exits non-zero for that path | Uploader logs `WARN: failed to push <path>`, increments `pathsFailed`, moves on. No in-run retry.       |
| Network blip / DNS hiccup                     | Same                                    | Same.                                                                                                   |
| Uploader process dies unexpectedly            | Some paths after the crash never upload | Post step detects dead PID, logs the count of unprocessed queue lines as a warning, does not fail.      |
| Post step skipped (job cancel/timeout)        | In-flight uploads die with the runner   | Accepted. Next run rebuilds and re-pushes affected paths.                                               |
| Hook can't write to queue file mid-build      | Daemon prints to stderr, build continues | Setup creates the queue with 0666 perms up front; this should not happen in practice.                  |

### Setup-side failures (these *do* fail loud)

These represent broken configuration where the cache promise cannot be
kept; failing loud is the correct response.

- Cannot write signing key file with 0600 perms → fail.
- Cannot reload nix-daemon after editing nix.conf → fail. (Without the
  reload, the hook never fires and the user silently gets no caching.)
- R2 SigV4 smoke test (`HEAD` on the bucket) returns 4xx → fail.
- Required input missing or malformed → fail before any side effect.

### nix.conf hygiene

We never overwrite `/etc/nix/nix.conf`. Setup appends a block delimited
by markers (paths are illustrative; actual paths are under
`$RUNNER_TEMP/wispy/`):

```
# >>> wispy >>>
extra-substituters = s3://wispy-cache?endpoint=https://<account>.r2.cloudflarestorage.com&region=auto
extra-trusted-public-keys = wispy-foo-1:<base64-pubkey>=
secret-key-files = /home/runner/work/_temp/wispy/signing.key
post-build-hook = /home/runner/work/_temp/wispy/hook.sh
# <<< wispy <<<
```

Post step removes everything between the markers, idempotently. This is
safe under repeated runs (composite actions can be referenced multiple
times in one job — though we don't expect it) and friendly to
self-hosted runners that may reuse the same nix.conf across jobs.

### Trust boundary for forks

`skip-push: true` causes setup to omit `post-build-hook` and
`secret-key-files` from the nix.conf block. Pull is still configured.
The canonical workflow expression is:

```yaml
skip-push: ${{ github.event.pull_request && github.event.pull_request.head.repo.fork }}
```

A fork's PR can still benefit from cache hits but cannot poison the
cache.

### Platform scope

- Linux x86_64 and aarch64: supported.
- macOS: setup exits non-zero with a clear message. Adding macOS
  requires a different daemon-reload mechanism (launchd) and is
  deferred to a future version.
- Self-hosted runners: should work; the marker-based nix.conf editing
  makes us non-destructive. We don't promise daemon-reload semantics
  across every distro and will document it as "report issues."

## Forward compatibility — the Cloudflare Worker

The v2 goal is a Cloudflare Worker fronting R2 that adds smarter
caching (upstream mirroring of `cache.nixos.org`, namespacing, GC,
metrics, possibly a public mirror for OSS users). v1 is designed so
the Worker is a one-line change for users:

- **Substituter URL is configurable.** `extra-substituters` already
  exists as an input. Users add `https://cache.example.com` and remove
  the bucket-direct entry, no other workflow change.
- **Signing keys are user-owned and stable.** Trust survives the
  migration: clients verify Worker responses with the same
  `trusted-public-keys` they already have.
- **Push stays direct to R2 in v1.** When the Worker grows write-path
  features (dedupe, content-addressed remap, multi-tier storage), we
  add an optional `push-endpoint` input that overrides the `s3://`
  destination. v1 doesn't bake R2 specifics anywhere outside
  `src/r2.ts`'s URL construction.
- **The "make it a substituter" abstraction is the entire boundary.**
  Replacing the substituter URL is a swap, not a rewrite.

## Testing strategy

Three layers, each catching a different class of bug.

### 1. Lint and typecheck (CI gate)

- `eslint` and `tsc --noEmit` on all TypeScript.
- `shellcheck` on `scripts/hook.sh`.
- `verify-dist`: rebuild `dist/` and `git diff --exit-code dist/` to
  ensure the checked-in bundle matches `src/`.

### 2. Unit tests (CI gate)

`vitest` against pure modules:

- `src/nixconf.ts`: `applyWispyBlock` produces the expected diff;
  `removeWispyBlock` is idempotent and touches only marker-delimited
  content; the round trip `apply` then `remove` returns the original.
- `src/inputs.ts`: missing required inputs produce a clear error
  before any side effect.
- Queue parser (inside `src/uploader.ts`, extracted): handles paths
  with spaces, comments, the sentinel, and EOF without sentinel.

### 3. Integration (CI gate, on a real R2 test bucket)

`.github/workflows/integration.yml`:

- **Job A — push:** build a trivial test flake from `tests/flake.nix`
  that produces an output salted with `${{ github.run_id }}` (so the
  store path is unique per run). Assert via the AWS SDK against R2's
  S3 endpoint that the expected `.narinfo` and `.nar.xz` objects
  exist.
- **Job B — pull:** depends on Job A, fresh runner, build the same
  flake. Assert via `nix path-info --json` that the output's `ca` /
  `signatures` field is populated and via runtime check that the build
  was substituted (`nix build` completes in well under the time the
  derivation takes to actually build from source).

A dedicated test bucket and dedicated test signing keys live in repo
secrets, scoped to this repo only.

```
├── src/
│   ├── main.ts                      # entry: main step

│   ├── nixconf.ts                   # pure helpers: applyWispyBlock, removeWispyBlock

```

Notes:

- `dist/` is committed. The `verify-dist` CI job guarantees it matches
  `src/`. Release flow is: tag → `dist/` is already current → done.
  Floating `v1` tag follows the standard GitHub Action convention.
- The hook is the only shell script and stays tiny. Everything else
  is TypeScript so it gets the benefit of types, tests, and one set
  of lint rules.
- `node_modules` is gitignored; `package-lock.json` is committed.
